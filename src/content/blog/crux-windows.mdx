*Draft — not final. May be revised.*

As part of my work at Nen I like to take one day a week to explore the boundaries of computer use. I've been following Arvind Narayanan's work on [AI as Normal Technology](https://www.normaltech.ai/) for a while now, and was pretty excited to read about [CRUX #1](https://cruxevals.com/crux-1). Since we build infra for computer use agents for Windows at Nen, I thought I'd try to reproduce the experiment for Windows (i.e. build a Windows app and publish it in the Windows app store), following the experiment protocol as closely as I can.

A few important differences:

- A mistake in the experimental setup meant that the agent had access to a compiled set of instructions to create the app. While I don't think this changes the outcome by much (the instructions were generated in a single pass which the experimental agent would have done anyway), it is worth calling out as a deviation from the experiment design.
  - In order to compensate for this, I wrote up a seperate set of instructions to reproduce this experiment. It turns out generalizing this wasn't trivial — going from an experimental methodology ("give agents some really hard and long running tasks") to experiment protocol ("make sure that a human is creating a new Gmail account, a Microsoft Developer account, set up properly sized VMs on the cloud etc.") to reproducible runs of an experiment ("we decided to store credentials in a plaintext file in the VM because these can be easily rotated if leaked, so make sure future runs retrieve them from this file") turned out to be a bigger project than expected — more to come here.
- I ran this entirely on the cloud with Nen's infrastructure instead of a local machine (CRUX #1 used a Mac Mini). Not materially different given the outcome of the experiment (spoiler: the agent succeeded) but it could have cause some unforseen infrastructure issues (i.e. IP blocking from specific cloud providers).
- Anthropic had already deprecated Opus 4.6 by the time I ran this. Since the point of the study is exploring the frontier of agent capability, this didn't invalidate the result of the study.

## The setup

Following CRUX #1's spirit as closely as I could: give the agent a real external gatekeeper, minimal instructions, a budget, and get out of the way. I used OpenClaw as the scaffold (CRUX #1 used a similar harness), used Nen's Windows infrastructure hosted on GCP, and gave it $500 of Anthropic budget and 14 days of wall-clock.

## What was reproduced

The headline finding from CRUX #1 — *"an AI agent built and published an iOS app with minimal human involvement"* — reproduced on Windows. TimeZonr is live in the Microsoft Store. Net human inputs during the entire run: two, both infrastructure (one heartbeat-rule fix, one budget-cap extension), zero on the app-building or Store-submission work itself.

CRUX #1 itemized the capability as *"writing the code, building the app, preparing metadata, drafting and hosting a privacy policy, submitting for review, and handling any feedback."* Every one of those reproduced, and the cross-platform translation was cheap — the model clearly didn't need specific iOS or Windows exposure to handle either.

## What was not reproduced

Interestingly, the original paper noted something my run did not reproduce at all: *"Partway through the evaluation, the agent changed its strategy to reduce monitoring cost significantly: it started using subagents rather than the entire context, and began using shorter daily memory files. This reduced the running cost from $35/hour to $3/hour."*

Nothing like this happened for me. My agent's waiting-day rate was ~$8/hour and it held that rate every idle day — never rethinking its heartbeat cadence, never spawning a subagent, never compacting its session. After four days I had spent ~$600 past the $500 cap before I noticed.

### Why didn't it happen?

The first hypothesis I reached for was my own: maybe my HEARTBEAT.md hid the budget signal. The rule I wrote said "sum `totalCost` for today, budget is $500 hard" — which the agent read literally, summing per-day instead of cumulative. Every day it saw $195 / $500 (≈39% of the cap) and reported all clear, every day the cumulative crept higher. No cost pressure → no reason to self-optimize. One-word bug, four-day consequence.

Reading the CRUX #1 page back more carefully undermines this neatly. They do not appear to have given their agent an explicit dollar cap at all — the agent self-optimized *without* budget awareness instruction, just from noticing the running cost was out of proportion to the work. So "my HEARTBEAT.md broke the signal" explains why *my* agent didn't react; it doesn't explain why *their* agent reacted without any signal.

## What I learned

One data point isn't a paper, and I've changed enough variables relative to CRUX #1 that direct comparison isn't honest. But a few things stood out:

1. **The framework is the interesting artifact.** The agent built a working Windows Store app; I already knew frontier models could do that. What I didn't know was where the scaffold design pressures you — and the answer is cache economics during waiting phases, not model capability.
2. **Minimal prompting survives real-world tasks.** The agent picked the concept, the framework (WinUI 3), the packaging strategy, the listing copy, and the recovery plan after rejection — from a bootstrap message of 30 characters and maybe five thousand words of personality/task/tool docs. The instinct to over-specify is strong and mostly wrong.
3. **Agents self-modify tooling, in a narrow sense, already.** The scp patch and the dexbox resolution bump aren't autonomous-research-lab escapes, but they're also not things I briefed the agent to do. It just read the error, read the code, and changed the code.

If you want to try to reproduce this — or a different version of it on a different platform — I wrote up the setup as a reusable methodology, with a two-agent pipeline (a "Designer" that generates the task-specific protocol and an "Operator" that runs it). That's a longer post, coming next.
